# 현업 프론트엔드 실무 확인사항 — 물음 세트

백엔드 경력자가 프리랜서 프론트엔드 작업에 들어가기 전, 현업에서 *실제로 어떻게 하는지*와 *아직 안 풀린 개념*을 확인하기 위한 질문 목록. 합류 후 각 물음을 현업에서 확인하고, 결과는 그 시점의 사례 연구나 관련 에픽으로 보낸다.

## 물음

### 1. 코드 컨벤션 유무·범위

*확인 대상:* ESLint/Prettier 룰셋, 팀 스타일 가이드의 존재와 강제 수준.

코드 컨벤션은 "코드를 어떤 모양으로 쓰자"는 팀의 약속이다. 들여쓰기, 따옴표, 변수 이름 짓는 법, import 정렬 같은 것들. 프론트에선 이걸 ESLint(문법·실수를 잡아주는 도구)와 Prettier(코드 모양을 자동으로 정리해주는 도구)로 강제하는 경우가 많다. 확인하고 싶은 건 세 갈래다 — 이런 도구가 깔려 있어서 저장만 하면 자동으로 맞춰지는가, 규칙이 문서로만 있고 사람이 지켜야 하는가, 아니면 아예 없는가. 백엔드의 Checkstyle처럼 익숙한 개념이지만 도구 종류도 다르고 "경고로만 띄우는지 / 에러로 막는지" 엄격함도 팀마다 달라서, 첫날 내 코드가 리뷰에서 반려되지 않으려면 미리 알아야 한다.

### 2. 퍼블리셔 분업·CSS 작성 주체

*확인 대상:* 마크업·CSS를 누가 쓰는가, CSS 방법론(BEM·Tailwind·CSS-in-JS).

화면의 겉모습(HTML 마크업과 CSS 스타일)을 누가 만드느냐의 문제다. 회사에 따라 "퍼블리셔"라는 역할이 따로 있어서 디자인을 HTML/CSS로 옮겨주고, 개발자는 그 위에 동작(데이터 연결·이벤트 처리)만 붙이기도 한다. 반대로 개발자가 마크업·CSS까지 다 짜는 곳도 많다. 후자라면 CSS를 어떤 방식으로 쓰는지도 봐야 한다 — 클래스 이름 규칙을 정해 쓰는 BEM, 미리 만들어진 유틸 클래스를 조합하는 Tailwind, 자바스크립트 안에서 스타일을 쓰는 CSS-in-JS 등. 백엔드 출신이 가장 부담스러워하는 영역이라, 내가 어디까지 책임지는지와 무엇을 미리 공부해 둘지를 이 질문으로 가른다.

### 3. 단위테스트의 한계

*확인 대상:* 단위테스트가 못 잡는 결함 유형, 통합·상호작용 경계.

백엔드에선 서비스 메서드 하나에 입력을 넣고 결과를 확인하는 식으로 단위테스트가 잘 들어맞았다. 프론트는 다르다. 화면이 제대로 그려졌는지, 버튼을 눌렀을 때 기대한 반응이 나오는지, 데이터를 불러오는 동안 로딩 표시가 맞게 뜨는지처럼 "눈에 보이는 동작"과 "여러 조각이 함께 맞물리는 상황"에서 버그가 난다. 이런 건 단위테스트로 다 못 잡는다. 그래서 알고 싶은 건, 단위테스트가 못 잡는 결함이 어떤 유형인지, 그래서 현업이 그 빈틈을 무엇으로 메우는지다 — 여러 화면을 실제로 클릭해보는 E2E 테스트, 사람이 직접 확인하는 수동 테스트 등. 어디까지 자동 테스트로 막고 어디부턴 다른 방법이 필요한지 선을 긋기 위함.

### 4. 상태 관리 라이브러리 선택 기준

*확인 대상:* Context / Zustand / Redux를 가르는 현업 잣대.

"상태"는 화면 곳곳에서 함께 쓰는 데이터다 — 로그인한 사용자 정보, 장바구니 내용, 다크모드가 켜졌는지 여부 같은 것. 이걸 컴포넌트들이 어떻게 나눠 갖게 할지 도와주는 도구가 상태 관리 라이브러리다. 후보가 여럿이다 — React에 기본으로 들어있는 Context API(간단하지만 규모가 커지면 불편), 가볍고 쉬운 Zustand, 크고 복잡한 앱에서 많이 쓰는 Redux. 무엇을 고를지는 앱 규모·복잡도·팀이 원래 쓰던 것에 따라 갈린다. 현업이 무엇을 기준으로 고르는지 알아야, 합류한 프로젝트가 왜 그걸 썼는지 이해하고 새 기능을 만들 때 과하게 또는 모자라게 설계하지 않는다.

### 5. API 통신 패턴

*확인 대상:* axios/fetch + React Query·SWR 등 데이터 패칭·캐싱 관행.

백엔드일 때는 내가 API를 만들어 *주는* 쪽이었다. 프론트에선 그 API를 *받아서* 화면에 뿌리는 쪽이 된다. 단순히 요청을 보내는 것(axios·fetch 같은 도구)만이 아니라, 받은 데이터를 잠시 저장해 다시 안 불러오게 하는 캐싱, 불러오는 중 로딩 표시, 실패했을 때의 에러 처리까지 한 묶음으로 다뤄야 한다. 요즘은 이걸 React Query나 SWR 같은 라이브러리가 대신 해준다. 현업이 데이터를 받아오는 층을 어떻게 짜는 게 표준인지 확인해서, 화면마다 fetch를 직접 덕지덕지 붙이는 식의 초보 구조를 피하려는 것.

### 6. 폴더 구조·아키텍처 컨벤션

*확인 대상:* feature-based 등 규모별 구조.

파일을 어떤 폴더에 어떻게 나눠 담느냐의 규칙이다. 백엔드는 controller·service·repository처럼 "역할별"로 나누는 게 익숙하다. 프론트는 기능(화면) 단위로 묶는 feature-based 구조를 많이 쓴다 — 예를 들어 "로그인"과 관련된 화면·로직·API를 한 폴더에 모으는 식. 앱이 작으면 단순하게, 크면 더 잘게 나눈다. 규모별로 어떻게 나누는지 알아야 새 파일을 만들 때 어디 둘지 매번 고민하지 않고, 처음 보는 코드베이스에서도 길을 빨리 찾는다.

### 7. 디자인 협업·핸드오프

*확인 대상:* Figma → 코드 전환, 디자인 토큰·컴포넌트 매핑.

디자이너가 그린 화면이 어떻게 내 코드로 넘어오는지의 절차다. 보통 Figma라는 디자인 도구로 화면을 만들고, 개발자는 거기서 색·간격·폰트 크기 같은 값을 보고 코드로 옮긴다. 이때 "디자인 토큰"(색·간격 같은 값에 이름을 붙여 디자인과 코드가 같은 기준을 쓰게 하는 것)이나, 디자인의 한 덩어리가 코드의 어떤 컴포넌트에 대응하는지 같은 약속이 있기도 하다. 디자이너와 함께 일하는 게 처음이라, 무엇을 디자이너에게 받고 무엇을 내가 알아서 해석·구현해야 하는지 경계를 알아야 서로 부딪히지 않는다.

### 8. 접근성 (웹 접근성·WCAG)

*확인 대상:* 시맨틱 마크업·스크린리더·키보드 내비게이션·ARIA를 어디까지 요구하는가.

접근성은 눈이 안 보이는 사람, 마우스를 못 쓰고 키보드만 쓰는 사람 등 모두가 화면을 쓸 수 있게 만드는 것이다. 예를 들어 이미지에 설명 글(alt)을 달아 화면을 읽어주는 프로그램(스크린리더)이 읽게 하고, 버튼을 키보드 탭으로 이동·실행할 수 있게 하고, 의미에 맞는 HTML 태그(버튼은 button 태그로)를 쓰는 것. WCAG는 이걸 어디까지 지켜야 하는지 정한 국제 기준이다. 백엔드에선 다룰 일이 거의 없던 영역이라, 현업이 이걸 법으로 꼭 지켜야 하는 수준으로 보는지(공공·금융은 의무인 경우가 많다) 아니면 여력 되는 대로 하는지를 알아야 마크업을 어떤 기준으로 쓸지 정해진다.

### 9. 성능 최적화

*확인 대상:* 렌더 성능(불필요한 리렌더·React.memo·useMemo), 번들 크기·코드 스플리팅·lazy loading, Core Web Vitals.

화면이 빠르고 가볍게 뜨도록 다듬는 일이다. 프론트의 성능은 백엔드와 결이 다르다. 대표적으로 세 가지다 — (1) 불필요한 다시 그리기 줄이기. React는 데이터가 바뀌면 화면을 다시 그리는데, 안 바뀌어도 되는 부분까지 자꾸 그려지면 느려진다(React.memo·useMemo로 막는다). (2) 사용자가 받는 파일 크기 줄이기. 코드를 화면별로 쪼개서 필요한 것만 먼저 내려받게 하는 코드 스플리팅·lazy loading. (3) 실제 체감 속도 지표인 Core Web Vitals(첫 화면이 뜨는 속도 등). 확인할 건, 현업이 이런 성능을 "번들 크기가 얼마 넘으면 빌드 실패" 같은 식으로 자동 검사(CI)로 강제하는지 여부.

### 10. 반응형·크로스브라우저 대응

*확인 대상:* 지원 브라우저·디바이스 범위, 미디어 쿼리·반응형을 누가 짜는가.

같은 화면을 휴대폰·태블릿·데스크톱 등 크기가 다른 화면에서도 깨지지 않게 만드는 게 반응형이고, 크롬·사파리·엣지 등 브라우저마다 다르게 보이지 않게 맞추는 게 크로스브라우저 대응이다. 백엔드에선 신경 쓸 일이 없던, 프론트만의 부담이다. 확인할 건 두 가지다 — 어디까지 대응해야 하는지(지원하는 브라우저·기기 범위를 팀이 정해둔다), 그리고 화면 크기에 따라 레이아웃을 바꾸는 작업(미디어 쿼리·반응형)을 퍼블리셔가 해주는지 내가 해야 하는지. 2번(CSS를 누가 쓰나)과 이어지는 질문.

### 11. 렌더링 모델·메타프레임워크

*확인 대상:* 순수 SPA인가, Next.js 같은 SSR/메타프레임워크인가.

화면을 "어디서" 그리느냐의 차이다. 순수 SPA는 브라우저가 빈 페이지를 받은 뒤 자바스크립트로 화면을 그린다. Next.js 같은 메타프레임워크는 서버에서 화면을 어느 정도 미리 그려서 보내준다(SSR). 이 선택이 라우팅 방식, 데이터를 언제 어디서 불러오는지, 폴더 구조까지 통째로 바꾼다. 그래서 합류한 프로젝트가 둘 중 무엇을 쓰는지를 가장 먼저 확인해야, 나머지 물음들(API 통신·폴더 구조 등)의 답도 거기에 맞춰 읽을 수 있다.
