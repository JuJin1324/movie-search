# 컴포넌트의 본질 — 의미를 가진 커스텀 태그

> [Your First Component](https://ko.react.dev/learn/your-first-component) 페이지를 읽고 정리한 감상과, 그에 대한 평가.

## 나의 감상

- 리액트는 시멘틱 HTML을 **커스텀한 HTML 태그**로 구현한 느낌이다. 이전에 시멘틱 HTML을 밀었지만 기존 관습(`<div>`만 쓰던 관성) 때문에 불가했던 걸, 리액트라는 프레임워크로 해결한 느낌.
- 이미지에 `Profile`이라는 의미를 부여했다.

## 평가

**핵심 직관은 정확하다.** 시멘틱 HTML이 `<div>` 범벅("div soup")을 벗어나 `<header>`·`<nav>`·`<article>`처럼 의미가 드러나는 태그를 쓰자는 흐름이었지만, 표준 태그 집합이 고정돼 있고 관습이 강해 실무에선 한계가 있었다. React 컴포넌트는 이를 다른 층위에서 푼다 — **의미 단위를 내가 정의해서 태그처럼 쓴다.** `<Profile />`, `<MovieCard />`, `<SearchBar />`는 이름만 봐도 무엇인지 읽힌다. "시멘틱 HTML의 정신을 프레임워크가 잇는다"는 비유는 잘 짚은 것이다.

```tsx
// div 범벅 대신
<div className="profile"><img .../><div className="name">...</div></div>

// 의미를 가진 태그로
<Profile />
```

`<img>` 하나에 `Profile`이라는 이름을 붙여, 단순 이미지 태그가 "이건 프로필이다"라는 **의미 단위**가 되었다는 관찰도 정확하다.

**다만 두 가지를 더하면 직관이 더 단단해진다.**

1. **컴포넌트는 결국 HTML로 렌더링된다.** `<Profile />`도 브라우저에는 `<img>` 같은 실제 표준 태그로 전달된다. 즉 의미는 *개발자/코드 층위*의 것이지, 브라우저가 `<Profile>`을 의미 있는 태그로 인식하는 건 아니다. 시멘틱 HTML과는 작동하는 층위가 다르다.

2. **그래서 "컴포넌트 = 시멘틱 태그"는 완전히 같지 않다.** 컴포넌트는 의미 부여를 넘어 **구조 + 동작 + 상태**(props·핸들러[01](01-component-declaration-and-handlers.md)·`useState`)를 함께 묶는 재사용 단위다. 반대로 SEO·접근성에 필요한 진짜 시멘틱 의미는 컴포넌트 *이름*이 아니라, 그 **안에서 실제로 어떤 HTML 태그를 쓰는지**가 결정한다. 그래서 `<Profile />` 안에서도 `<header>`·`<article>` 같은 시멘틱 태그는 별도로 챙겨야 한다.

**정리:** 감상은 컴포넌트를 "의미 단위로 사고하게 해주는 도구"로 본 점에서 본질을 잘 잡았다. 여기에 "그 의미는 코드 층위의 것이고, 브라우저용 시멘틱 의미는 컴포넌트 안에서 따로 책임진다"는 경계만 더하면 오해 없이 완성된다.
