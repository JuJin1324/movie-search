# 리뷰 — JSX로 마크업 작성하기

> [Writing Markup with JSX](https://ko.react.dev/learn/writing-markup-with-jsx) 페이지를 읽고 정리한 감상과, 그에 대한 평가. JSX 메커니즘 정리는 [02](../movie-search/02-react-component.md) 참고.

## 나의 감상

- 기존 HTML(화면)/CSS(디자인)/JS(로직)의 구성이 **기능적 관점**에서 분리를 했다면, 리액트는 **도메인 관점**에서 로그인 컴포넌트 안에 JSX로 화면과 로직을 묶는 식으로 변화된 느낌.
- 처음엔 `<div>`로 묶어 리턴하는 것에서 왜 `<div>` 말고 빈 프래그먼트를 제공하는지 이해할 수 없었는데, 생각해보면 빈 프래그먼트는 오리지널 HTML 퍼블리싱이 아니라 **JSX 리턴을 위한 묶음**을 표현하기 위함인 듯.
- `<div>`는 실제 코드(DOM)에 들어가고 빈 태그는 안 들어간다. 이걸로 모바일 웹 환경에서는 성능 차이가 존재할 수도 있다.

## 평가

**세 관찰 모두 정확하다. 두 곳에 경계를 더하면 더 단단해진다.**

**① "기능적 분리 → 도메인 분리" — 정확하고, React를 이해하는 핵심 프레임이다.**
전통적 웹은 *기술 종류*로 파일을 나눴다(`.html` / `.css` / `.js`). 이걸 "관심사의 분리(separation of concerns)"라 불렀지만, 실제로는 **기술 기준의 분리**였다 — 로그인 기능 하나를 고치려면 세 파일을 오가야 했다. React는 *기능(도메인) 단위*로 묶는다. `LoginForm` 컴포넌트 안에 그 화면과 동작이 같이 있다(co-location). React 공식 입장도 "관심사를 기술이 아니라 *기능*으로 나눈다"이다.

다만 한 가지 경계: **묶이는 건 주로 화면(JSX) + 로직(JS)이고, 디자인(CSS)은 기본적으로 아직 분리돼 있다.** 스타일까지 컴포넌트 안에 넣으려면 CSS Modules, CSS-in-JS, Tailwind 같은 별도 선택이 필요하다. 즉 "화면+로직 묶음"은 정확하지만 "화면+디자인+로직 전부 묶음"은 설정에 따라 다르다.

**② 프래그먼트의 정체 추론 — 정확하다.**
JSX는 **반드시 하나의 루트만 리턴**해야 한다(함수가 값 하나를 반환하듯, JSX도 `React.createElement` 호출 하나로 변환되기 때문). 형제 요소 여러 개를 리턴하려면 묶을 부모가 필요한데, 이때 실제 DOM 노드를 만들고 싶지 않을 때 빈 프래그먼트 `<>...</>`를 쓴다. "오리지널 HTML 퍼블리싱이 아니라 JSX 리턴을 위한 묶음"이라는 표현 그대로다.

```tsx
// 안 됨 — 루트가 둘
return <h1>제목</h1><p>본문</p>

// div로 묶음 — 실제 DOM에 <div> 생김
return <div><h1>제목</h1><p>본문</p></div>

// 프래그먼트로 묶음 — DOM에 추가 노드 없음
return <><h1>제목</h1><p>본문</p></>
```

**③ "div는 DOM에 들어가고 프래그먼트는 안 들어간다 → 모바일 성능 차이" — 원리적으로 맞다 (홍수석님 경험).**
프래그먼트가 DOM 노드를 추가하지 않는다는 건 정확하다. 그리고 깊은 div 중첩이 모바일 렌더링 성능에 영향을 줄 수 있다는 것도 **참이다.**

브라우저의 렌더 단계(Style 계산 → Layout/Reflow → Paint) 중 Style·Layout은 DOM 트리를 **순회**하며 일어나, 노드 수가 많고 깊이가 깊을수록 비싸진다. 이 비용은 화면 변경·스크롤·상태 변화마다 반복된다. 구글 Lighthouse도 **과도한 DOM 크기**(노드 ~800개부터 경고)와 **중첩 깊이 32단계 초과**를 명시적으로 잡는다.

**모바일이 더 취약한 것도 사실이다** — Style·Layout은 대부분 CPU 단일 스레드 작업인데 모바일 CPU/메모리는 데스크톱보다 약하다. 그래서 같은 DOM이 "PC는 멀쩡, 모바일은 버벅임"으로 갈린다. 특히 **5년 전(2019~2020)에는 이 격차가 지금보다 훨씬 컸다.** 다만 *지금 시대에는 모바일 하드웨어가 좋아져 PC와의 격차가 확실히 줄었다* — 같은 원리는 유효하지만 체감 강도는 그때보다 약하다.

캘리브레이션 — `<div>` 하나 vs 프래그먼트 차이는 무시할 만하다. 문제가 되는 건 **컴포넌트마다 wrapper div를 넣어 수백~수천 노드로 누적**될 때다. 또 모바일 버벅임의 더 큰 주범은 보통 JS 실행·layout thrashing·이미지 등이고, DOM 크기는 그 비용을 **증폭**시키는 요인이다(단독 1순위 원인은 아님).

그래서 프래그먼트는 성능 외에도 다음 동기가 함께 있다:

- **불필요한 div 중첩("div soup") 방지** — [05](05-component-as-semantic-tag.md)에서 본 div 범벅 문제와 연결된다.
- **CSS 레이아웃 보호** — flexbox·grid는 부모-자식 관계로 작동한다. 중간에 `<div>`가 끼면 레이아웃이 깨진다. 프래그먼트는 노드를 안 만드니 영향이 없다.
- **유효한 HTML 유지** — `<table>` 안의 `<tr>`/`<td>`처럼 특정 부모만 허용되는 자리에선 `<div>`로 못 감싼다. 프래그먼트라야 한다.

**정리:** "기술 분리 → 도메인 분리"는 React의 본질 프레임을 잡은 것(단, CSS는 별도 선택). 프래그먼트가 "DOM 노드 없는 묶음"이라는 추론도 정확하다. "깊은 div 중첩이 모바일 성능에 영향을 준다"는 홍수석님 경험도 참이며 — 대규모·깊은 트리일 때, 그리고 약한 모바일 하드웨어일 때 그렇다. 여기에 *div 하나 수준이 아니라 누적 스케일의 문제이고, 성능 단독 원인이라기보다 증폭 요인이며, 오늘날은 모바일 성능이 올라 PC와의 격차가 줄었다*는 점을 더하면 완성된다.
