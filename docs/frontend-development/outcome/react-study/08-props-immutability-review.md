# 리뷰 — props의 불변성

> [Passing Props to a Component](https://ko.react.dev/learn/passing-props-to-a-component) 페이지를 읽고 정리한 감상과, 그에 대한 평가. props 전달 메커니즘·단방향 흐름 정리는 [movie-search/02](../movie-search/02-react-component.md), 핸들러로 부모 상태를 바꾸는 패턴은 [01](01-component-declaration-and-handlers.md) 참고.

## 나의 감상

- props를 읽으며 가장 크게 깨달은 건 **props의 불변 성질**이다.
- 매개변수를 불변으로 두면서 **상태(state)와 분리**한다. 그래서 props가 임의로 변경되지 않게 방지되고, 이것이 **코드의 신뢰**를 높이는 역할을 한다.
- 적절한 **DTO의 관례**라는 생각이 들었다. 받는 쪽이 읽기만 하는 데이터 묶음이라는 점에서 그렇다.
- 이 부분에서 라이브러리(React) 설계를 참 잘했다는 생각이 들었다.

## 평가

**핵심 직관은 정확하다 — "props는 읽기 전용"은 React가 데이터 흐름을 예측 가능하게 만드는 바로 그 설계 원칙이다.** 여기에 세 가지 경계를 더하면 "왜 신뢰가 올라가는가"가 끝까지 설명된다.

**① "불변 → 상태와 분리 → 신뢰" — 방향이 정확하고, React 공식 입장 그대로다.**
React 문서도 "컴포넌트는 자신의 props를 절대 바꿔서는 안 된다(props are read-only)"고 못 박는다. 분리의 구도는 이렇다 —

| | state | props |
|---|---|---|
| 소유자 | **자기 자신**이 가진 데이터 | **부모**가 내려준 입력 |
| 변경 권한 | 자신이 `useState`로 바꿈 | 자신은 **못 바꿈**(읽기 전용) |
| 성격 | 가변(mutable) | 불변(read-only) |

자식 입장에서 props는 "내가 만든 게 아니라 받아 쓰는 입력"이라, 함부로 고치면 안 되는 게 자연스럽다. 이 분리 덕분에 "이 값이 어디서 바뀌는가?"의 답이 **항상 소유자 한 곳**으로 좁혀진다 — 신뢰가 올라가는 진짜 이유가 이것이다.

**② DTO 관례 비유 — 적절하다. 다만 "전송용 DTO"보다 "불변 값 객체(immutable value object)"에 더 가깝다.**
[movie-search/02](../movie-search/02-react-component.md)에서 props를 Command DTO에 대응시킨 것과 같은 직관이다. 호출자(부모)가 데이터를 담아 넘기고 받는 쪽(자식)은 읽기만 한다는 구조가 정확히 겹친다. 한 가지만 좁히면 — DTO는 보통 *경계를 넘는 전송*에 방점이 있지만, props의 불변성은 *받은 값을 그 안에서 못 고친다*는 데 방점이 있다. 그래서 Java로 치면 `record`나 `final` 필드만 가진 불변 값 객체에 가장 가깝다.

**③ "임의로 변경되지 않게 방지" — 의미는 맞지만, 한 칸 정밀하게: React가 *막는* 게 아니라 *규약*으로 약속한 것이다.**
여기가 "라이브러리 설계를 잘했다"는 감상에 붙일 가장 중요한 캘리브레이션이다.

- **언어/런타임이 강제하지 않는다.** props로 넘어온 객체는 JS에서 `Object.freeze`로 동결되지 않는다. 자식에서 `props.title = "x"`를 쓰면 기술적으로는 실행된다(다만 해선 안 된다). TypeScript의 구조분해나 `readonly`도 *컴파일 타임 신호*일 뿐 런타임 잠금이 아니다.
- **그래서 불변성은 "잠금"이 아니라 "단방향 데이터 흐름이라는 모델 + 규약"으로 보장된다.** React가 천재적인 건, 강제 장치 없이도 *재렌더링 방식* 자체로 불변성이 자연스럽게 지켜지게 만든 점이다(아래 ④).

이 구분이 "설계를 잘했다"를 더 정확하게 만든다 — React는 props를 *못 바꾸게 자물쇠를 채운 게 아니라*, 바꿀 이유가 없도록 흐름을 설계했다.

**④ 놓치기 쉬운 한 조각 — props는 "영원히 안 바뀌는 값"이 아니라 "렌더마다 새로 받는 스냅샷"이다.**
"불변"이라는 말 때문에 *값이 고정된다*고 오해하기 쉬운데, 실제로는 반대다. 부모의 데이터가 바뀌면 부모가 **재렌더링**되고, 자식에게 **새 props를 다시 내려준다**. 즉 props는 *수정(mutate)*되는 게 아니라 매 렌더마다 *교체(replace)*된다.

```tsx
// 부모의 state가 바뀜 → 부모 재렌더 → 자식은 새 title을 새 props로 받음
<MovieCard title={query} />   // query가 바뀌면, 자식은 "바뀐 props"가 아니라 "새 props"를 받는다
```

그래서 정확한 그림은 **"각 렌더 안에서 props는 불변(고정된 스냅샷)이고, 렌더와 렌더 사이에는 새 값으로 갈아끼워진다"**이다. 이 조합(불변 스냅샷 + 재렌더)이 바로 신뢰의 메커니즘이다 — 렌더링이 `(props, state) → UI`라는 **순수 함수**처럼 동작하게 되고, 같은 입력이면 같은 화면이 보장된다.

**⑤ 그럼 자식이 부모 데이터를 바꿔야 할 땐? — 값이 아니라 함수를 내려받는다.**
불변이라고 자식이 부모에 영향을 못 주는 건 아니다. 부모가 *변경 함수*를 prop으로 내려주고(`onChange`, `onSave`), 자식은 그 함수를 호출만 한다 — 데이터를 직접 고치는 게 아니라 "바꿔달라고 요청"하는 구조다. 이 패턴은 [01](01-component-declaration-and-handlers.md)의 `handle*`/`on*` 컨벤션, [movie-search/02](../movie-search/02-react-component.md)의 단방향 흐름과 그대로 이어진다. 불변성과 콜백 prop은 한 쌍이다.

**정리:** "props 불변 → 상태와 분리 → 신뢰"라는 감상은 React 설계의 본질을 정확히 잡았고, DTO 관례라는 비유도 적절하다. 여기에 세 경계를 더하면 완성된다 — (1) 불변성은 언어가 *강제*하는 잠금이 아니라 단방향 흐름이라는 *규약*으로 지켜진다(그래서 "설계를 잘했다"가 더 정확해진다), (2) DTO보다 *불변 값 객체*에 가깝다, (3) props는 *고정된 값*이 아니라 *렌더마다 갈아끼우는 스냅샷*이며, 이 "불변 스냅샷 + 재렌더" 조합이 렌더링을 순수 함수로 만들어 신뢰를 만든다. 자식이 부모를 바꿔야 할 땐 값 대신 콜백 함수를 내려받는다.
