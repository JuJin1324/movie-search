# Plan Stories

## 목적

해결 방안 탐색(explore-solutions)이 그린 *TO-BE 한 장*을 받아, *실행 가능한 Story 목록*으로 분해한다.

쉽게 말하면 — **TO-BE는 설계도, Stories는 공정표**. 같은 건물을 가리키지만 답하는 질문이 다르다.

| TO-BE가 답하는 것 | Stories가 답하는 것 |
|---|---|
| 완성된 모습은 어떤가 | 어떤 순서로 만들 것인가 |
| 무엇이 어떻게 연결되는가 | 한 단계가 얼마나 걸리는가 |
| 어디가 비어 있는가 | 끝났다는 걸 무엇으로 아는가 (완료 기준) |
| | 도중에 무엇을 결정해야 하는가 (ADR) |
| | 무엇은 *나중에 결정*인가 (보류) |
| | 만들면서 *추가로 합의해야 할 것*은 무엇인가 (운용 규칙 등) |

## 입력

해결 방안 탐색(explore-solutions) 산출물 — TO-BE 다이어그램 + 흐름 + 컴포넌트.

## 출력

`epics/<epic-slug>/3-plan/stories.md` *초안*. 전체 흐름과 Story별 상세가 한 자리에 정리된 Story 목록 문서.

### 템플릿

```markdown
# {문제 제목} — Story 목록

선행: [../2-explore/explore-solutions.md]

## 전체 흐름

`mermaid 다이어그램 — Story 간 순서·의존관계`

## Story 1 — {제목}

### 목적
### 실행 완료 기준
### ADR 후보 (있으면)

## Story 2 — {제목}
...

## Story N (보류) — {제목}

### 목적
### 보류 이유
### 트리거 (어떤 조건에서 꺼내는가)

```

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 Story 단위·완료 기준을 채워간다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 전체 흐름을 먼저 합의한 뒤, *Story 단위로* 한 개씩 사용자와 주고받으며 보충한다.

1. **전체 흐름 다이어그램** — TO-BE에서 Story 후보를 뽑아 mermaid로 순서·의존관계를 그린 뒤, *왜 이 순서인지*를 설명하고 확인받는다. Story 입자 크기(*1일 / 1주 / 1사이클 후 판단* 등)와 *빠진/불필요한 Story 없는지*도 이 단계에서 합의.

2. **Story별 상세** — Story 1부터 순서대로 *한 Story씩* 작성한다. 각 Story마다: AI가 *목적 / 실행 완료 기준 / ADR 후보*를 먼저 적고, 사용자가 *보충하고 싶은 자리*를 짚으면 그 자리에서 갱신한다. 다음 Story로 넘어가기 전에 합의.

## 경계

- **문제 재정의·해결 방안 탐색은 여기서 안 한다.** 작업 중 TO-BE가 흔들리면 explore-solutions로 돌아간다.
- **Task 분해는 여기서 안 한다.** 한 Story를 검토 가능한 Task 목록으로 나누는 일은 plan-tasks의 자리.
- **첫 사이클의 회고를 미리 적지 않는다.** Story의 *실행 완료 기준*에 *마찰 메모*는 들어가지만, 그건 *다음 사이클 입력*이지 완성된 회고가 아니다.
- **TO-BE에 없던 자리가 Stories에서 떠오르는 건 정상.** 이건 TO-BE가 *부족했다*는 신호가 아니라 plan-stories가 *공정표 자리*를 한다는 신호. 다만 그 자리가 *해결 방향 자체*를 흔들면 explore-solutions로 돌아가야 한다 — 운용 규칙·보류·ADR 같은 *공정표 차원의 추가*만 여기서 처리.
