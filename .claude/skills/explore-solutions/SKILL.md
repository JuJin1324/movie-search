# Explore Solutions

## 목적

문제 정의(define-problem)가 끝낸 문제를 받아 *해결된 모습(TO-BE)*을 한 장의 그림으로 그린다. 문제 정의가 *지금 이 문제가 어떻게 발생하는지(AS-IS)*를 보여줬다면, explore-solutions는 *이렇게 바뀌면 문제가 풀린다*를 같은 결로 보여준다.

## 입력

- 문제 정의(define-problem) 산출물
- 문제 정의 단계에서 인박스로 넘어온 해결 아이디어 시드 (있으면)

## 출력

`epics/<epic-slug>/2-explore/explore-solutions.md` *초안*. 다음 3단 패턴 + 미결정 자리로 정리된 해결 방향 문서를 만든다.

- **TO-BE 다이어그램** — 해결된 모습의 *에픽 수준의 구조·흐름*을 한 장으로 보이는 mermaid 그림
- **흐름 설명** — 다이어그램이 보여주는 흐름을 글로 풀어 설명
- **컴포넌트 설명** — 다이어그램에 등장하는 각 구성요소의 역할
- **미결정** — 아직 안 정한 것, 다음 사이클로 넘기는 질문 (있으면)

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 *어디를 어떻게 바꾸면 풀리는지*를 한 장으로 보게 되고, 문서는 그 이해의 흔적이다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 떠오른 아이디어 시드를 한 번 정리한 뒤, 3단 패턴을 한 자리씩 채운다.

**시드 정리 (사전 단계)** — 문제 정의에서 넘어온 인박스와 작업 중 떠오른 아이디어를 짧게 모아 보여주고, 어떤 시드가 다이어그램의 재료가 될지 사용자와 합의한다. 이 단계는 가볍게, 시드를 *나열*하는 수준으로 유지한다 — 본 작성은 다이어그램부터.

1. **TO-BE 다이어그램** — 해결된 모습을 mermaid 다이어그램으로 그린 뒤, 어떤 구조로 그렸고 왜 그렇게 그렸는지 설명하고 확인받는다. 다이어그램은 *문제 정의의 AS-IS와 같은 결*로 그린다 — 같은 노드들이 어떻게 바뀌었는지, 빠진 자리가 어떻게 채워졌는지가 보이게.
2. **흐름 설명** — 1번에서 합의된 다이어그램의 흐름을 글로 풀어 적은 뒤, 어떻게 이해했고 왜 그렇게 적었는지 설명하고 확인받는다.
3. **컴포넌트 설명** — 다이어그램의 각 구성요소를 *한 컴포넌트씩* 같은 패턴(작성 → 이해/이유 설명 → 확인)으로 진행한다.

작성 도중 실행 계획(어떻게 쪼개 만들지)이 떠오르면 별도 인박스에 옮겨 두고 컴포넌트로 다시 돌아온다 — 그건 plan-stories의 자리.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **문제 재정의는 여기서 안 한다.** 작업 중 문제 정의 자체가 흔들리면 define-problem으로 돌아간다.
- **실행 계획은 여기서 안 한다.** 어떻게 쪼개 만들지는 plan-stories의 자리.
- **다이어그램에 미결정·대안 분기를 넣지 않는다.** 한 시점의 *합의된 TO-BE 한 장*만 담는다. 미결정은 *미결정* 섹션으로.
