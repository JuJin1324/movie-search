# Model Scenarios

## 목적

구조와 이해관계자가 흐름 안에서 빠짐없이 작동하는지 *대표 시나리오*로 검증한다. 모델링 단계 안에서 model-structure·model-stakeholders와 *서로를 보강하는 루프*로 작동한다. 도메인에 따라 이 셋 중 어디서 시작해도 되며, 셋이 정합되어야 모델링이 닫힌다.

## 입력

structure.md·stakeholders.md 초안(있다면) 또는 도메인의 핵심 사용 사례 후보. 시나리오 검증 트리거(흐름 변경, 이해관계자 변경, 새 사용 사례 등장 등).

## 출력

`modeling/scenarios.md` *초안*. 도메인의 *대표 시나리오*를 시퀀스로 그려, structure와 stakeholders가 실제 흐름 안에서 빠짐없이 작동하는지 검증할 수 있게 한다. 각 시나리오는 다음 패턴을 따른다:

- **시퀀스 다이어그램** — 시나리오의 시간축 위 통신 흐름을 한 장으로 보이는 mermaid 그림
- **흐름 설명** — 시퀀스를 글로 풀어 설명 (한 단락)
- **흐름 특유의 모양** — 이 시나리오에서 드러나는 구조적 특성 (몇 개 불릿)
- **단계별 상세** — 시퀀스의 각 단계가 *어떤 이해관계자 책임을 실행하는지* 매핑

초안에는 *가장 대표적인 시나리오 하나*까지만 위 패턴을 다 채우고, 나머지 후보는 *제목과 한 줄 요약*만 둔다. 시나리오 목록을 채우는 일은 사용자와의 대화로 진행.

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 완성한다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 시나리오 단위로 작성하며, 매 시나리오는 *시퀀스 → 흐름 설명 → 모양 → 단계별 상세* 4단 패턴을 따른다.

1. **시나리오 후보 합의** — 도메인의 대표 사용 사례 후보를 사용자와 합의해 시나리오 목록을 잡는다.
2. **대표 시나리오 채움** — 첫 시나리오 하나를 4단 패턴으로 끝까지 채운다. 시퀀스부터 그리고, 설명·모양·표는 그 위에 매단다.
3. **시나리오 풍부화** — 같은 핑퐁으로 나머지 시나리오를 한 시나리오씩 채워간다. 매번 AI가 시퀀스 후보를 제안하고 사용자가 확인·수정·검증한다.
4. 시나리오들을 다 채우면 전체를 다시 훑어 *구조·이해관계자와 정합되는지* 점검. 모순이 드러나면 model-structure·model-stakeholders로 루프백.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **구조 정의는 여기서 안 한다.** 작업의 단계 흐름·단위 계층을 정의하는 자리는 model-structure.
- **이해관계자 정의는 여기서 안 한다.** 새 이해관계자가 시나리오에서 드러나면 model-stakeholders로 보낸다.
- **시나리오 풍부화는 *루프 안*에서 일어난다.** 시나리오 검증이 구조·이해관계자의 빈 곳을 드러내면 그 쪽으로 돌아간다.
- **문제 정의·해결 방안 탐색·실행은 여기서 안 한다.** 모델링은 *토대*만 정의한다.
