# Define Problem

## 목적

승급된 백로그 항목 하나를 받아 *지금 이 문제가 어떻게 발생하는지(AS-IS)*를 한 장의 그림으로 보여준다. 입력이 드러내고 있지 않은 숨은 구조를 사용자와 함께 끌어낸다.

뒤이은 explore-solutions가 *해결된 모습(TO-BE)*을 같은 결로 그린다 — 두 자리는 *AS-IS / TO-BE* 한 쌍으로 평행을 이룬다.

## 입력

승급된 백로그 항목 하나. 백로그 표의 수준 칼럼이 무엇이든, 정의 시점엔 *에픽 슬롯*(`epics/<epic-slug>/`)으로 다뤄진다.

## 출력

`epics/<epic-slug>/1-define/define-problem.md` *초안*. 다음 3단 패턴으로 정리된 문제 정의 문서를 만든다.

- **AS-IS 다이어그램** — 현재 문제가 발생하는 *에픽 수준의 구조·흐름*을 한 장으로 보이는 mermaid 그림
- **흐름 설명** — 다이어그램이 보여주는 흐름을 글로 풀어 설명
- **컴포넌트 설명** — 다이어그램에 등장하는 각 구성요소의 역할

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 *자기 문제를 더 깊이 이해*해 가고, 문서는 그 이해의 흔적이다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 입력을 받자마자 *입력 분류*를 거친 뒤, 3단 패턴을 한 자리씩 채운다.

**입력 분류 (사전 단계)** — 사용자 입력을 *문제 부분*과 *해결 부분*으로 갈라서 보여주고 확인받는다. 문제 부분은 그대로 다음 단계의 재료가 되고, 해결 부분은 explore-solutions 인박스로 즉시 옮겨 둔다. 사용자 입력에 해결 부분이 섞여 들어오는 경우가 많으므로 이 단계를 생략하지 않는다.

1. **AS-IS 다이어그램** — 승급된 백로그 항목을 바탕으로 mermaid 다이어그램을 그린 뒤, 어떤 구조로 이해했고 왜 그렇게 그렸는지 설명하고 확인받는다. 다이어그램은 *현재 문제가 발생하는 에픽 수준의 구조(AS-IS)*만 담는다. 해결 방안·개입 지점·해결 후 모습 같은 노드는 금지 — 그건 explore-solutions(TO-BE)의 자리. 작성 도중 해결 아이디어가 떠오르면 별도 인박스에 옮겨 두고 다이어그램으로 다시 돌아온다.
2. **흐름 설명** — 1번에서 합의된 다이어그램의 흐름을 글로 풀어 적은 뒤, 어떻게 이해했고 왜 그렇게 적었는지 설명하고 확인받는다.
3. **컴포넌트 설명** — 다이어그램의 각 구성요소를 *한 컴포넌트씩* 같은 패턴(작성 → 이해/이유 설명 → 확인)으로 진행한다.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **백로그 항목의 추가·분류·승급 판단은 여기서 안 한다.** 그건 organize-backlog의 자리. define-problem은 *이미 승급된 한 항목*을 받는다.
- **해결 방안 탐색은 여기서 안 한다.** 정의 작업 중 떠오른 해결 아이디어는 explore-solutions로 넘긴다.
- **실행 계획은 여기서 안 한다.** 어떻게 쪼개 실행할지는 plan-stories의 자리.
