# Organize Backlog

## 목적

프로젝트의 *에픽 후보 풀*을 채우고 분류·정리한다. 모델링 결과 또는 외부 사유(아이디어·요청·메모)로 들어온 항목을 백로그 자리에 합의·반영해 다음 사이클의 입력으로 준비한다.

## 입력

다음 중 하나 이상:
- 모델링 산출물(structure.md / stakeholders.md / scenarios.md) — 에픽 후보 도출 단서
- 사용자가 제시한 외부 사유 — 외부 요청·아이디어·중단 메모 등 modeling과 무관한 항목
- 기존 백로그 — 분류·우선순위·표현 정리 트리거

## 출력

프로젝트의 `epics/backlog.md` *초안*. 에픽 후보 풀의 단일 파일. 에픽 후보를 한 표로 모아둔 큐.

| 제목 | 수준 | 맥락 | 상태 |
|---|---|---|---|
| {한 줄 제목} | 프로젝트 / 에픽 / 스토리 / 태스크 | {사유 또는 맥락} | 대기 / 진행 중 / 보류 / 완료 |

- **제목** — 다룰 문제의 이름. 한 줄로 내용까지 표현한다. 상세는 백로그에 두지 않고 define-problem으로 끌어올렸을 때 채운다.
- **수준** — 항목의 작업 수준. 프로젝트 / 에픽 / 스토리 / 태스크 중 하나.
- **맥락** — 어떤 사유로 들어왔는지 (modeling 결과 / 외부 요청 / 아이디어 / 메모 등). 선택.
- **상태** — 항목의 라이프사이클. 대기 / 진행 중 / 보류 / 완료.
- **우선순위** — 별도 칼럼 없이 *행 순서*로 관리. 위에 있을수록 높은 우선순위. 다음 사이클 입력은 *위에서부터*.

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 채워간다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 항목 단위로 추가/수정/제거를 합의한다.

1. **트리거 분류** — 입력이 모델링 변경인지, 외부 사유인지, 누적 항목 정리인지 사용자와 확인.
2. **백로그 항목 후보 제시** — AI가 백로그에 *추가/수정/제거*할 항목을 한 줄씩 제시한다.
3. **부분 합의·반영** — 사용자가 항목 단위로 확인·검증해 백로그 표에 반영.
4. **상태·행 순서 정리** — 항목이 어느 정도 모이면 *상태 갱신·행 순서 조정*을 같은 핑퐁으로 진행. 상태는 라이프사이클을, 행 순서는 우선순위를 표현.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **문제 정의는 여기서 안 한다.** 에픽 후보의 한 줄 제목까지만. AS-IS 구조 정의는 define-problem의 자리.
- **해결 방안 탐색은 여기서 안 한다.** 백로그 항목에 해결 아이디어가 섞여 들어오면 *문제 부분만* 백로그에 두고 해결 부분은 별도로 둔다.
- **모델링은 여기서 안 한다.** 모델링 결과를 *읽어* 에픽 후보를 도출할 뿐, 모델링 산출물을 직접 수정하지 않는다 — 그건 model-* 스킬의 자리.
- **비공식 입구를 두지 않는다.** 외부 사유로 들어온 항목도 백로그 *자리 자체*에 organize-backlog를 통해 반영한다. "임시 메모장" 같은 우회 통로를 두지 않는다.
