# Model Structure

## 목적

프로젝트의 작업이 어떤 단계를 거치고 어떤 단위로 분해되는지 정의한다. 모델링 단계 안에서 model-stakeholders·model-scenarios와 *서로를 보강하는 루프*로 작동한다. 도메인에 따라 이 셋 중 어디서 시작해도 되며, 셋이 정합되어야 모델링이 닫힌다.

## 입력

새 프로젝트의 초기 정의 또는 기존 프로젝트의 갱신 트리거(흐름 변경, 단위 추가, 스킬 추가 등).

## 출력

`modeling/structure.md` *초안*. 도메인에 맞춰 *복수의 3단 블록*으로 구성된 구조 문서를 만든다. 각 블록은 다음 패턴을 따른다:

- **다이어그램** — 한 주제를 한 장으로 보이는 mermaid 그림
- **다이어그램 설명** — 그림이 보여주는 흐름·관계를 글로 풀어 설명
- **각 컴포넌트 설명** — 다이어그램에 등장하는 각 구성요소의 정의·역할

블록의 *주제*는 도메인에 따라 달라진다. 일반적으로 *작업의 단계 흐름*과 *작업 단위 계층*은 거의 모든 도메인에 등장하며, 그 위에 도메인 특화 블록(예: process-engineering이면 스킬 구조·파일 관리·경계 선언)이 더해진다.

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 완성한다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 블록 단위로 작성하며, 매 블록은 *다이어그램 → 설명 → 컴포넌트 상세* 3단 패턴을 따른다.

1. 도메인에 맞는 블록 후보(주제)를 사용자와 합의한다.
2. 각 블록을 순서대로 핑퐁으로 채운다. 다이어그램부터 그리고, 설명과 컴포넌트는 그 위에 매단다.
3. 블록을 다 채우면 전체를 다시 훑어 일관성을 확인한다.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **이해관계자 정의는 여기서 안 한다.** 시스템과 통신하는 주체와 그들의 책임을 정의하는 자리는 model-stakeholders.
- **시나리오 검증은 여기서 안 한다.** 구조와 이해관계자가 흐름 안에서 빠짐없이 작동하는지 검증하는 자리는 model-scenarios.
- **문제 정의·해결 방안 탐색·실행은 여기서 안 한다.** 모델링은 *토대*만 정의한다. 한 문제를 다루는 일은 5단계의 나머지(문제 정의·탐색·계획·실행)가 맡는다.
