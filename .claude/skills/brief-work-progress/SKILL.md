# Brief Work Progress

## 목적

Git 기록과 작업 트리를 단서로 *일이 어디까지 왔는지*와 *어디서부터 이어가면 되는지*를 복원한다.

정규 흐름(define → explore → plan → execute) 바깥에 있는 *횡단 스킬*. 어느 단계에서 떠났든, 세션이 바뀌어 돌아왔을 때 *재진입 지점*을 잡아준다. 커밋을 설명하는 스킬이 아니라 *진행 중인 일*을 설명하는 스킬.

## 입력

다음 중 하나 이상:
- **사용자 트리거** — "어디까지 했지", "다음 뭐부터", "인수인계해줘" 같은 재개·인계 요청
- **Git 상태** — 최근 커밋·작업 트리·브랜치 (상태 추론 단서)
- **산출물 디렉터리** — `epics/<epic-slug>/` 아래 정의·탐색·계획·실행 산출물(`1-define/` / `2-explore/` / `3-plan/`), 결정 문서, TODO

## 출력

*채팅 브리핑*. 파일로 저장하지 않는다 — 다른 component와 달리 디렉터리에 산출물을 남기지 않는 것이 의도적인 경계.

브리핑은 다음 자리를 가진다:

- **어디까지 왔나** — 완료된 작업 단위와 현재 상태
- **남은 것** — 아직 비어 있거나 결정·검증이 필요한 부분
- **어디서부터 하면 되나** — 다음에 열 파일과 시작 행동
- **정리할 것** — 미커밋 변경, 흐름이 섞인 파일, 커밋 전 확인 지점
- **참고 경로** — 본문에서 부른 문서들의 풀 경로 모음

산출물 자체가 완성본이 아니다 — 사용자가 브리핑을 보고 다음 스킬(plan-tasks / execute-task 등)을 호출하기 위한 *진입점*만 제공.

## 작동 방식

조사 → 브리핑. 코드는 수정하지 않는다.

1. **Git으로 최근 흐름 찾기** — `git status`, 최근 커밋 몇 개로 어떤 작업 흐름이 움직였는지 후보를 잡는다.
2. **최근 커밋에서 맥락 추출** — `git show`로 어떤 문제/Story/Task가 진행되었고 어떤 산출물이 갱신되었는지 읽는다.
3. **산출물에서 실제 진행 상태 확인** — 커밋이 가리키는 문제 디렉터리의 plan/execute 산출물을 직접 읽어 *완료된 단위*와 *비어 있는 자리*를 확인. 이 단계가 브리핑의 핵심.
4. **커밋 이후 변경 확인** — 작업 트리가 깨끗하지 않으면 staged/unstaged/untracked를 별도로 읽어 *최근 커밋 이후 진행 중인 작업*을 분리.
5. **브리핑 작성** — 채팅에 직접 출력. 첫 문장은 커밋 정보가 아니라 *현재 작업 상태*로 시작.

## 경계

- **코드는 수정하지 않는다.** 브리핑만 한다. 수정이 필요하면 사용자가 다음 스킬을 호출.
- **새 계획은 여기서 안 한다.** 다음 행동은 *재개 지점*까지만 가리킨다. Task 분해는 plan-tasks, Story 분해는 plan-stories의 자리.
- **커밋 요약 스킬이 아니다.** 브리핑의 주어는 커밋이 아니라 *일*. 커밋 메시지를 그대로 반복하지 않는다.
- **`HEAD` 커밋과 미커밋 변경을 섞지 않는다.** "완료됨"과 "진행 중"의 경계가 흐려지면 재개 지점이 흔들린다.
- **파일로 남기지 않는다.** 다른 component의 산출물이 *디렉터리에 누적되는 정본*이라면, brief-work-progress의 산출물은 *그 자리에서 소비되는 채팅 브리핑*. 진행 상태는 Git과 산출물에 이미 있고, 별도 상태 파일을 두면 그것도 동기화 대상이 되어 침식이 시작된다.
