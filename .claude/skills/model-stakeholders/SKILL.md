# Model Stakeholders

## 목적

시스템과 통신하는 이해관계자와 그들의 *핵심 기대*를 정의한다. 모델링 단계 안에서 model-structure·model-scenarios와 *서로를 보강하는 루프*로 작동한다. 도메인에 따라 이 셋 중 어디서 시작해도 되며, 셋이 정합되어야 모델링이 닫힌다.

## 입력

새 프로젝트의 초기 정의 또는 기존 프로젝트의 이해관계자 갱신 트리거(새 청중 추가, 책임 변경, 외부 시스템 결합 등).

## 출력

`modeling/stakeholders.md` *초안*. 도메인에 맞춰 *복수의 3단 블록*으로 구성된 이해관계자 문서를 만든다. 각 블록은 다음 패턴을 따른다:

- **다이어그램** — 이해관계자와 시스템의 통신 관계를 한 장으로 보이는 mermaid 그림 (C4 Context 결)
- **다이어그램 설명** — 이해관계자 분류·관계의 결을 글로 풀어 설명
- **각 컴포넌트 설명** — 다이어그램에 등장하는 각 이해관계자의 역할과 *핵심 기대 한 줄*

각 이해관계자의 기대는 *한 줄*까지만 초안에 담는다. 처음부터 많은 기대를 추정해 채우지 않는다 — 완성본은 여러 기대 목록이 될 수 있지만, 그것은 사용자가 AI 에이전트와 대화하며 채워가는 결과물이다.

산출물 자체가 완성본이 아니다 — 사용자가 AI 에이전트와 대화하며 완성한다. 스킬은 *진입점*과 *초안*을 제공할 뿐.

## 작동 방식

핑퐁. 블록 단위로 작성하며, 매 블록은 *다이어그램 → 설명 → 컴포넌트 상세* 3단 패턴을 따른다.

1. 도메인의 이해관계자 후보(직접 통신·간접 영향 등)를 사용자와 합의한다.
2. 각 블록을 순서대로 핑퐁으로 채운다. 다이어그램부터 그리고, 설명과 각 이해관계자 한 줄 기대는 그 위에 매단다.
3. **이해관계자 기대 풍부화** — 한 줄 기대로 초안을 닫은 뒤, 같은 핑퐁으로 사용자와 *기대 목록*을 한 항목씩 늘려간다. **각 기대 항목은 '기대 — 왜 그 기대가 정당화되는가'를 한 줄에 함께 박는다.** 근거(왜)는 사용자가 가진 현실 제약에서 끌어온다. 매번 AI가 추가 후보를 제안하고 사용자가 확인·수정·추가로 키운다. 처음부터 다 채우지 않고, 사용자가 시스템을 더 깊이 이해하면서 함께 자란다.
4. 블록을 다 채우면 전체를 다시 훑어 일관성을 확인한다.

매 단계마다 *작성 → 이유 설명 → 사용자 확인* 순서.

## 경계

- **구조 정의는 여기서 안 한다.** 작업의 단계 흐름·단위 계층을 정의하는 자리는 model-structure.
- **시나리오 검증은 여기서 안 한다.** 구조와 이해관계자가 흐름 안에서 작동하는지 검증하는 자리는 model-scenarios.
- **이해관계자 기대를 풍부하게 채우는 일은 여기서 안 한다.** 초안은 한 줄까지만 — 나머지는 사용자와의 대화로 완성.
- **문제 정의·해결 방안 탐색·실행은 여기서 안 한다.** 모델링은 *토대*만 정의한다.
